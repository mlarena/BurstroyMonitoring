# BurstroyMonitoring.Worker

Фоновый сервис (Worker Service) для автоматического сбора и обработки данных с дорожных метеостанций и датчиков мониторинга в системе BurstroyMonitoring.

## Основные функции

*   **Автоматический опрос**: Периодическая проверка базы данных на наличие постов, требующих опроса (интервал проверки — настраивается пользователем).
*   **Параллельная обработка**: Одновременный опрос нескольких постов и датчиков для обеспечения высокой производительности.
*   **Поддержка типов оборудования**:
    *   **DSPD**: Состояние дорожного полотна (температура, слой осадков, коэффициент сцепления).
    *   **IWS**: Метеорологические параметры (ветер, влажность, давление, тип осадков).
    *   **MUEKS**: Технологический мониторинг (питание, АКБ, температура шкафа, датчик открытия).
    *   **DOV**: Видимость и освещенность.
    *   **DUST**: Концентрация пыли.
*   **Диагностика**: Логирование HTTP-статусов, времени отклика и ошибок парсинга данных в базу данных и файлы логов.

## Технологический стек

*   **.NET 10.0** (Worker Service)
*   **СУБД** (PostgreSQL)
*   **Serilog** (Логирование в консоль и файлы)

## Обработка данных и работа с БД

### 1. Получение и разбор (парсинг) ответов
Сервис ожидает от датчиков ответы в формате JSON. Процесс обработки включает:
*   **Валидация формата**: Проверка корректности структуры JSON. Если датчик прислал некорректный ответ (например, HTML-страницу ошибки вместо данных), сервис зафиксирует ошибку парсинга.
*   **Извлечение метаданных**: Из каждого ответа автоматически извлекается серийный номер устройства (поле `Serial`). Если он отличается от того, что указан в базе, система обновит его автоматически.
*   **Преобразование типов**: Данные от разных типов датчиков (DSPD, IWS, MUEKS и др.) распределяются по соответствующим категориям. Числовые значения, приходящие в виде строк, автоматически конвертируются в числа.
*   **Временные метки**: Сервис анализирует время замера, переданное самим датчиком (`DataTime`), и приводит его к единому формату UTC для корректного отображения в отчетах.

### 2. Запись результатов в базу данных
Информация сохраняется в несколько этапов для обеспечения полной прослеживаемости:
*   **Сессии опроса**: Каждая попытка опроса поста фиксируется как отдельная сессия. В ней хранится общее количество датчиков, время начала/окончания и итоговое количество успешных ответов.
*   **Сырые результаты**: В таблицу `SensorResults` сохраняется полный текст ответа от датчика (ResponseBody), HTTP-статус и время отклика в миллисекундах. Это позволяет администраторам увидеть "первоисточник" данных в случае спорных ситуаций.
*   **Структурированные данные**: Основные параметры (температура, ветер, состояние дороги и т.д.) записываются в специализированные таблицы. Это обеспечивает высокую скорость построения графиков и отчетов в пользовательском интерфейсе.
*   **Журнал ошибок**: Если датчик недоступен по сети или произошла ошибка при записи в БД, подробное описание проблемы (включая текст исключения) сохраняется в таблицу ошибок. Это позволяет технической поддержке оперативно выявлять неисправное оборудование.

## Структура проекта

*   `Worker.cs`: Главный цикл управления расписанием опроса.
*   `Services/`:
    *   `SensorPollingService.cs`: Сетевое взаимодействие с датчиками (HTTP/HTTPS).
    *   `DataProcessingService.cs`: Парсинг JSON-ответов и бизнес-логика обработки данных.
    *   `DatabaseService.cs`: Взаимодействие с БД (сохранение результатов, управление сессиями).
    *   `LoggerService.cs`: Специализированное логирование ошибок оборудования в БД.
    *   `ConfigurationService.cs`: Управление настройками приложения.

```json
{

```

## Жизненный цикл опроса

1.  **Поиск**: Выборка активных постов, время опроса которых наступило.
2.  **Сессия**: Регистрация новой сессии опроса в таблице `PollingSessions`.
3.  **Запрос**: Параллельная отправка HTTP-запросов к датчикам.
4.  **Парсинг**: Преобразование JSON-ответов в модели данных (DSPD, IWS и др.).
5.  **Сохранение**: Запись структурированных данных и метаданных ответа в БД.
6.  **Ошибки**: В случае сбоя (таймаут, 404, ошибка парсинга) информация фиксируется в `SensorErrors`.
