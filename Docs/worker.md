# Руководство пользователя
## BurstroyMonitoring.Worker - Сервис мониторинга датчиков

---

## 1. Общие сведения

**BurstroyMonitoring.Worker** — это фоновый сервис (Windows Service / Linux daemon) для автоматического опроса и сбора данных с различных типов датчиков мониторинга. Сервис работает по расписанию, опрашивает датчики через HTTP API и сохраняет полученные данные в базу данных PostgreSQL.

### 1.1 Основные функции

| Функция | Описание |
|---------|----------|
| Автоматический опрос датчиков | HTTP GET запросы к датчикам с заданным интервалом |
| Параллельная обработка | Одновременный опрос до 100 датчиков |
| Автоматические повторные попытки | До 3 попыток при ошибках сети |
| Обработка различных типов датчиков | DSPD, IWS, MUEKS, DOV, DUST |
| Сохранение данных в PostgreSQL | Все показания сохраняются в БД |
| Логирование ошибок | В файлы и базу данных |
| Динамическая конфигурация | Настройки можно изменять через БД без перезапуска |

### 1.2 Поддерживаемые типы датчиков

| Тип | Назначение | Сохраняемые данные |
|-----|------------|-------------------|
| **DSPD** | Дорожная станция | Температура дороги, состояние покрытия, коэффициент сцепления, GPS координаты |
| **IWS** | Метеостанция | Температура, влажность, давление, скорость/направление ветра, осадки, CO2 |
| **MUEKS** | Станция мониторинга воздуха | Напряжение, ток, температура бокса, видимость |
| **DOV** | Датчик видимости | Дальность видимости |
| **DUST** | Датчик пыли | PM1, PM2.5, PM10 (взвешенные частицы) |

## 2. Системные требования

### 2.1 Минимальные требования

| Компонент | Требование |
|-----------|------------|

| ОЗУ | 512 MB (рекомендуется 1 GB) |

## 3. Установка и настройка

### 3.1 Установка

**Windows:**
```cmd
dotnet publish -c Release -o C:\Services\BurstroyMonitoring.Worker
```

**Linux:**
```bash
dotnet publish -c Release -o /opt/burstroy-monitoring/worker
```

### 3.2 Файл конфигурации (`appsettings.json`)

Основной файл конфигурации находится в корневой директории приложения:

```json
{
  "ConnectionStrings": {
    "DefaultConnection": "Host=localhost;Database=sensordb;Username=postgres;Password=12345678"
  },
  "Logging": {
    "FilePath": "logs/burstroy-worker-.log",
    "FileRetainedFileCountLimit": 30,
    "ConsoleMinimumLevel": "Information",
    "FileMinimumLevel": "Information"
  },
  "Polling": {
    "MaxConcurrentTasks": 100,
    "HttpTimeoutSeconds": 30,
    "RetryCount": 3,
    "RetryDelayMs": 1000
  },
  "SaveResponseBody": {
    "Default": true
  }
}
```

### 3.3 Параметры конфигурации

#### Подключение к БД (`ConnectionStrings`)

| Параметр | Описание | Пример |
|----------|----------|--------|
| `DefaultConnection` | Строка подключения к PostgreSQL | `Host=localhost;Database=sensordb;Username=postgres;Password=***` |

#### Логирование (`Logging`)

| Параметр | Описание | Значение по умолчанию |
|----------|----------|----------------------|
| `FilePath` | Путь к файлам логов | `logs/burstroy-worker-.log` |
| `FileRetainedFileCountLimit` | Количество хранимых файлов логов | `30` |
| `ConsoleMinimumLevel` | Уровень логирования в консоль | `Information` |
| `FileMinimumLevel` | Уровень логирования в файл | `Information` |

**Уровни логирования (от наименьшего к наибольшему):**
`Verbose` → `Debug` → `Information` → `Warning` → `Error` → `Fatal`

#### Опрос датчиков (`Polling`)

| Параметр | Описание | Значение по умолчанию |
|----------|----------|----------------------|
| `MaxConcurrentTasks` | Максимальное количество параллельных запросов | `100` |
| `HttpTimeoutSeconds` | Таймаут HTTP запроса (секунды) | `30` |
| `RetryCount` | Количество повторных попыток при ошибке | `3` |
| `RetryDelayMs` | Задержка между повторными попытками (мс) | `1000` |

#### Сохранение тела ответа (`SaveResponseBody`)

| Параметр | Описание | Значение по умолчанию |
|----------|----------|----------------------|
| `Default` | Сохранять ли тело ответа по умолчанию | `true` |

## 4. Управление через базу данных

Сервис поддерживает динамическую конфигурацию через таблицу `WorkerConfigurations`. Изменения в этой таблице применяются автоматически (интервал обновления — 60 секунд).

### 4.1 Таблица конфигурации (`WorkerConfigurations`)

| Колонка | Тип | Описание |
|---------|-----|----------|
| `Key` | string | Ключ настройки |
| `Value` | string | Значение настройки |
| `IsActive` | boolean | Активна ли настройка |

### 4.2 Доступные настройки в БД

| Ключ настройки | Описание | Пример значения |
|----------------|----------|-----------------|
| `Polling.MaxConcurrentTasks` | Макс. параллельных запросов | `50` |
| `Polling.RetryCount` | Количество повторных попыток | `5` |
| `Polling.RetryDelayMs` | Задержка между попытками (мс) | `2000` |
| `Polling.HttpTimeoutSeconds` | Таймаут HTTP (сек) | `45` |
| `Configuration.RefreshIntervalSeconds` | Интервал обновления конфигурации (сек) | `120` |
| `SaveResponseBody.Default` | Сохранять тело ответа по умолчанию | `true` |
| `SaveResponseBody.Sensor.{id}` | Сохранять тело ответа для конкретного датчика | `false` |

### 4.3 Примеры SQL запросов для настройки

```sql
-- Изменить максимальное количество параллельных запросов
INSERT INTO "WorkerConfigurations" ("Key", "Value", "IsActive") 
VALUES ('Polling.MaxConcurrentTasks', '50', true)
ON CONFLICT ("Key") DO UPDATE SET "Value" = '50';

-- Отключить сохранение тела ответа для датчика с ID=10
INSERT INTO "WorkerConfigurations" ("Key", "Value", "IsActive") 
VALUES ('SaveResponseBody.Sensor.10', 'false', true)
ON CONFLICT ("Key") DO UPDATE SET "Value" = 'false';

-- Увеличить таймаут HTTP до 60 секунд
INSERT INTO "WorkerConfigurations" ("Key", "Value", "IsActive") 
VALUES ('Polling.HttpTimeoutSeconds', '60', true)
ON CONFLICT ("Key") DO UPDATE SET "Value" = '60';
```

### 4.4 Приоритет настроек

Настройки применяются в следующем порядке приоритета:

1. **База данных** (таблица `WorkerConfigurations`) — наивысший приоритет
2. **appsettings.json** — используется, если настройка не найдена в БД
3. **Значение по умолчанию** — встроенное в код

## 6. Запуск и остановка

### 6.1 Запуск в консоли (для отладки)

```bash
dotnet BurstroyMonitoring.Worker.dll
```

### 6.2 Установка как Windows Service

```cmd
sc create "BurstroyMonitoringWorker" binPath="C:\Services\BurstroyMonitoring.Worker\BurstroyMonitoring.Worker.exe" start=auto
net start BurstroyMonitoringWorker
```

### 6.3 Установка как systemd сервиса (Linux)

Создайте файл `/etc/systemd/system/burstroy-worker.service`:

```ini
[Unit]
Description=Burstroy Monitoring Worker
After=network.target postgresql.service

[Service]
Type=simple
WorkingDirectory=/opt/burstroy-monitoring/worker
ExecStart=/usr/bin/dotnet /opt/burstroy-monitoring/worker/BurstroyMonitoring.Worker.dll
Restart=always
RestartSec=10
User=burstroy

[Install]
WantedBy=multi-user.target
```

Затем выполните:

```bash
sudo systemctl daemon-reload
sudo systemctl enable burstroy-worker
sudo systemctl start burstroy-worker
```

### 6.4 Остановка сервиса

**Windows:**
```cmd
net stop BurstroyMonitoringWorker
```

**Linux:**
```bash
sudo systemctl stop burstroy-worker
```

## 7. Логирование и мониторинг

### 7.1 Файлы логов

Логи сохраняются в директории `logs/` (относительно рабочей директории приложения):

| Файл | Содержимое |
|------|------------|
| `burstroy-worker-{YYYYMMDD}.log` | Основной лог приложения |
| `errors-{YYYYMMDD}.log` | Лог ошибок доступа к датчикам |

### 7.3 Расшифровка уровней логирования

| Уровень | Обозначение | Описание |
|---------|-------------|----------|
| Verbose | [VRB] | Детальная отладочная информация |
| Debug | [DBG] | Отладочная информация |
| Information | [INF] | Информационные сообщения (нормальная работа) |
| Warning | [WRN] | Предупреждения (проблемы, не требующие немедленного вмешательства) |
| Error | [ERR] | Ошибки (проблемы, требующие внимания) |
| Fatal | [FTL] | Критические ошибки (приложение остановлено) |

## 9. Устранение неполадок

### 9.1 Частые проблемы и решения

| Проблема | Возможная причина | Решение |
|----------|-------------------|---------|
| Сервис не запускается | Неправильная строка подключения к БД | Проверьте `ConnectionStrings` в `appsettings.json` |
| Ошибки подключения к датчикам | Сетевые проблемы, датчик недоступен | Проверьте доступность URL датчика, настройки сети |
| Высокая нагрузка на CPU | Слишком большое значение `MaxConcurrentTasks` | Уменьшите значение через БД |
| Ошибки таймаута | Датчик медленно отвечает | Увеличьте `HttpTimeoutSeconds` |
| Дублирование данных | Ошибка в логике опроса | Перезапустите сервис |
