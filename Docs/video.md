# Пользовательская документация
## BurstroyMonitoring.VideoMonitoring

### 1. Назначение приложения

Приложение **BurstroyMonitoring.VideoMonitoring** — это фоновый сервис (Windows Service / Linux daemon) для автоматического захвата кадров (snapshots) с IP-камер видеонаблюдения. Приложение работает в режиме мониторинга, периодически подключается к камерам по протоколу RTSP и сохраняет снимки в файловую систему, записывая информацию о них в базу данных PostgreSQL.

### 2. Системные требования

- **Операционная система**: Windows / Linux / macOS
- **.NET Runtime**: .NET 10.0 или выше
- **FFmpeg**: Утилита должна быть установлена и доступна в PATH системы
- **PostgreSQL**: Сервер базы данных версии 12 или выше
- **Дисковое пространство**: Не менее 10 ГБ (зависит от количества камер и частоты снимков)

Файл конфигурации `appsettings.json` находится в корневой директории приложения.

#### 4.1. Основные настройки

```json
{
  "ConnectionStrings": {

  },
  "SnapshotSettings": {
    "BaseDirectory": "C:/git/BurstroyMonitoring/BurstroyMonitoring.TCM/wwwroot/snapshots"
  }
}
```

#### 4.2. Настройка подключения к базе данных

Параметр `DefaultConnection` содержит строку подключения к PostgreSQL:

| Параметр | Описание | Пример |
|----------|----------|--------|
| Host | Адрес сервера БД | localhost, 192.168.1.100 |
| Port | Порт подключения | 5432 (стандартный) |
| Database | Имя базы данных | sensordb |
| Username | Имя пользователя | postgres |
| Password | Пароль | ваш_пароль |

#### 4.3. Настройка директории для снимков

Параметр `BaseDirectory` указывает путь, куда будут сохраняться снимки:

- **Linux**: `/var/www/snapshots`

**Важно**: Убедитесь, что у приложения есть права на запись в указанную директорию.

#### 4.4. Настройка логирования

Логирование настраивается в `Program.cs`:
- Логи выводятся в консоль
- Логи сохраняются в файлы `logs/worker-ГГГГММДД.log` (ежедневная ротация)
- Уровень логирования: Information

### 6. Принцип работы

1. Приложение запускается и подключается к базе данных PostgreSQL
2. Каждые 10 секунд выполняется цикл проверки всех камер
3. Для каждой камеры проверяется, когда был сделан последний снимок
4. Если интервал с момента последнего снимка превышает `PollingInterval` камеры, делается новый снимок
5. Снимок сохраняется в директорию: `{BaseDirectory}/{CameraId}/{GUID}.jpg`
6. Информация о снимке записывается в таблицу `Snapshots` БД

#### 6.1. Процесс захвата кадра

Приложение использует FFmpeg для захвата кадра:
```bash
ffmpeg -rtsp_transport tcp -i "rtsp://..." -frames:v 1 -q:v 2 "output.jpg" -y
```

**Параметры:**
- `-rtsp_transport tcp`: Использование TCP протокола для стабильности
- `-i`: RTSP URL камеры
- `-frames:v 1`: Захват только одного кадра
- `-q:v 2`: Качество снимка (2 — высокое, 31 — низкое)
- `-y`: Перезапись существующего файла

#### 6.2. Аутентификация камер

Если камера требует аутентификацию, приложение автоматически вставляет логин и пароль в RTSP URL:
- Исходный URL: `rtsp://192.168.1.100:554/stream`
- Результат: `rtsp://username:password@192.168.1.100:554/stream`

### 7. Мониторинг и логи

#### 7.1. Логирование

Логи записываются в два места:
1. **Консоль** (при запуске в интерактивном режиме)
2. **Файлы** в папке `logs/worker-YYYYMMDD.log`

Пример лога успешной работы:
```
[INFO] Worker started at: 2024-01-15 10:00:00
[INFO] Taking snapshot for camera: Входная группа
[INFO] Snapshot saved for camera: Входная группа
```

Пример лога ошибки:
```
[ERROR] Failed to take snapshot for camera: Задний двор
[WARN] Error in worker loop: Connection refused
```

#### 7.2. Проверка статуса

Для проверки работы приложения:

1. **Проверьте логи** на наличие ошибок
2. **Проверьте папку со снимками** — новые файлы должны появляться регулярно
3. **Проверьте базу данных** — таблица `Snapshots` должна пополняться новыми записями


### 8. Устранение неполадок

#### 8.1. Приложение не запускается

**Проблема**: Ошибка подключения к базе данных
**Решение**: 
- Проверьте, запущен ли PostgreSQL
- Проверьте строку подключения в `appsettings.json`
- Убедитесь, что указаны правильные логин и пароль

#### 8.2. Снимки не создаются

**Проблема**: Ошибка "ffmpeg not found"
**Решение**: 
- Установите FFmpeg
- Проверьте, что FFmpeg доступен в командной строке: `ffmpeg -version`
- Укажите полный путь к FFmpeg в коде (требуется модификация)

**Проблема**: Ошибка доступа к директории
**Решение**:
- Проверьте права на запись в `BaseDirectory`
- Создайте директорию вручную, если она не существует

**Проблема**: Не удаётся подключиться к камере
**Решение**:
- Проверьте, доступен ли RTSP URL через VLC Player
- Проверьте правильность логина и пароля
- Убедитесь, что камера поддерживает RTSP
- Проверьте сетевые настройки и файрвол

#### 8.3. Высокая нагрузка на CPU

**Проблема**: FFmpeg потребляет много ресурсов
**Решение**:
- Увеличьте `PollingInterval` для камер в базе данных
- Уменьшите качество снимков (измените параметр `-q:v` на 5-10)

### 9. Производительность и масштабирование

- **Частота проверки**: Приложение проверяет камеры каждые 10 секунд
- **Интервалы снимков**: Задаются индивидуально для каждой камеры в БД (поле `PollingInterval` в секундах)
- **Одновременные операции**: Снимки обрабатываются последовательно, без параллелизации
- **Хранение снимков**: Рекомендуется настроить автоматическую очистку старых снимков через внешний скрипт или задание планировщика

### 10. Безопасность

- Строка подключения к БД содержит пароль — защитите файл конфигурации
- Рекомендуется использовать **User Secrets** для разработки
- В production используйте переменные окружения или защищённое хранилище секретов
- RTSP пароли камер хранятся в базе данных — обеспечьте безопасность БД
